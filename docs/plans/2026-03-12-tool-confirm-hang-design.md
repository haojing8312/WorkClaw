# Tool Confirm Hang Design

**Problem**

部分工具型对话在返回少量文本后长期停留在 `thinking`。真实会话记录显示 run 只写入了 `run_started`，没有 `run_failed` 或 `run_completed`。结合运行代码，最可疑的路径是工具确认事件发出后，后端阻塞等待确认结果，导致整条执行链不返回。

**Confirmed Scope**

1. 只处理工具确认导致的长时间挂起。
2. 不修改模型路由策略和会话模型兜底逻辑。
3. 优先保证 run 一定结束，不再出现无限转圈。

**Chosen Design**

采用双端收口：

1. 后端将工具确认等待从长阻塞改为短超时。
2. 后端确认超时后显式将该工具视为拒绝，并继续收敛到可完成或可失败的 run。
3. 前端在会话切换、组件卸载或确认弹窗无法继续时，主动回传拒绝，避免后端遗留等待中的确认通道。
4. 只读/低风险工具继续依赖现有权限分类，不在本次额外扩展免确认规则。

**Why This Design**

- 能直接解决“永远转圈”的用户感知问题。
- 改动集中在工具确认链路，风险比调整模型/路由更低。
- 同时覆盖“前端没确认”和“后端等太久”两个真实故障面。

**Verification**

1. 后端单测覆盖确认超时后不会无限等待。
2. 前端测试覆盖收到确认事件后，在清理阶段自动回传拒绝。
3. 真实场景回归至少验证“查看当前工作目录”这类工具型消息不会再无限转圈。
