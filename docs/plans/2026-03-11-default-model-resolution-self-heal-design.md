# 默认模型统一解析与自愈设计

## 1. 目标

统一前后端“默认模型”的读取逻辑，避免不同入口各自按数组首项或 SQL 首项兜底，导致界面默认与实际执行不一致。

本次目标：

1. 所有前端入口统一通过 `is_default` 解析默认模型
2. 所有后端默认模型查询统一走同一套解析函数
3. 无默认标记但存在模型时，系统自动补一个默认
4. 用户优先感知到“始终有一个明确默认”，而不是长期依赖隐式首项

## 2. 当前问题

当前默认模型读取不一致：

1. `App.tsx` 多处直接使用 `models[0]?.id`
2. `ChatView.tsx` 显示当前模型时直接用 `models[0]`
3. 后端部分命令使用 `WHERE is_default = 1`，查不到再退回首条记录
4. 兜底逻辑散落在多个调用点，没有统一自愈

风险：

1. 列表中的“默认”标签已经切换，但实际新会话可能仍然取数组第一项
2. 如果数据异常导致没有默认标记，系统会默默 fallback，却不恢复数据一致性

## 3. 设计原则

1. 用户优先：有可用模型时尽量不报错
2. 单一规则：默认模型解析逻辑只维护一套
3. 自动自愈：发现无默认但有模型时，立即补默认
4. 渐进兼容：保留对旧数据/异常状态的兜底能力

## 4. 方案选择

### 方案 A：仅统一读取，不自愈

规则：

1. `find(is_default)`
2. 否则 fallback 第一条

优点：

1. 实现简单

缺点：

1. 异常状态会长期存在
2. 数据与行为仍可能持续不一致

### 方案 B：统一读取并自动自愈

规则：

1. 优先取默认标记
2. 查不到时 fallback 第一条可用模型
3. 同时把 fallback 选中的模型补写为默认

优点：

1. 用户无感可用
2. 系统状态会恢复为单一明确默认
3. 后续调用点更容易稳定

缺点：

1. 需要在后端解析函数中附带一次写操作

### 方案 C：无默认直接报错

优点：

1. 规则严格

缺点：

1. 明显不符合产品体验
2. 旧数据和异常删除后会直接影响用户使用

### 结论

采用方案 B。

## 5. 前端设计

新增统一 helper，建议文件：

- `apps/runtime/src/lib/default-model.ts`

提供：

1. `getDefaultModel(models)`
2. `getDefaultModelId(models)`

规则：

1. `models.find((item) => item.is_default)`
2. 否则 `models[0]`
3. 否则 `null`

替换范围：

1. `App.tsx` 中所有 `models[0]?.id`
2. `ChatView.tsx` 中当前模型显示使用的 `models[0]`

说明：

前端只负责统一读取，不负责自愈落库。自愈应由后端完成，避免多端状态竞争。

## 6. 后端设计

在 `apps/runtime/src-tauri/src/commands/models.rs` 中新增统一函数，建议：

1. `resolve_default_model_id_with_pool`
2. `resolve_default_model_config_with_pool`

行为：

1. 优先查 `is_default = 1`
2. 若无结果，则查首个非搜索模型
3. 若找到 fallback 项，则调用 `set_default_model_with_pool` 自动补默认
4. 返回最终选中的模型

过滤规则：

1. 仅处理非 `search_%` 模型
2. 若调用方要求必须有 API Key，则在查询条件中附加 `TRIM(api_key) != ''`

## 7. 使用点收敛

### 7.1 App 会话创建

以下入口改为调用前端 helper：

1. 普通新会话
2. 团队入口会话
3. 安装技能后自动创建会话
4. 员工直达会话
5. 员工助手入口

### 7.2 ChatView

“当前模型”显示改为使用前端 helper 返回的默认模型，而不是数组首项。

### 7.3 后端命令

以下逻辑改为复用统一解析函数：

1. 翻译默认模型解析
2. 员工群组执行创建会话时的默认模型解析
3. 其他已知直接查询默认模型并 fallback 首项的命令

## 8. 测试设计

### 8.1 前端测试

至少覆盖：

1. `list_model_configs` 返回“首项非默认、第二项默认”时，新会话使用第二项
2. `ChatView` 显示默认模型名，而不是数组首项模型名

### 8.2 后端测试

至少覆盖：

1. 有明确默认时返回该默认
2. 无默认但有模型时返回首项并自动补默认
3. 自愈后数据库中仅有一个默认模型
4. 可选：带 `api_key` 过滤时仍按相同规则工作

## 9. 结论

顶级产品体验不应在“默认缺失”时直接报错，也不应长期依赖静默 fallback。

最合适的路径是：

1. 所有调用点统一读取默认模型
2. 缺失默认时保持可用
3. 同时自动修复为“明确有一个默认”

这样既保证体验稳定，也保证系统状态长期一致。
