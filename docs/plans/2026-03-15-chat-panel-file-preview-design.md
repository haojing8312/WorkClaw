# 会话面板文件预览设计

**日期：** 2026-03-15

## 背景

当前会话右侧“面板”中的“文件”标签，已经具备工作空间文件列表、文件树、文件预览区，以及基础的 Markdown 和 HTML 双模式预览雏形：

- `WorkspaceFilesPanel` 负责文件树、搜索和选择状态
- `FilePreviewPane` 负责预览区渲染
- Tauri 命令 `list_workspace_files` / `read_workspace_file_preview` 负责文件枚举和读取

这意味着本次需求不是“从零实现文件预览”，而是把已有能力收敛成一个稳定、可上线的首版，并补齐适合实际使用的抽屉宽度和文件列表可读性。

## 目标

在延续现有文件预览链路的前提下，优先实现并稳定以下能力：

- `.md` 文件预览
- `.html` 文件预览
- 面板可调宽抽屉
- 长文件名可读且类型明确

首版范围内要求：

- Markdown 支持“渲染预览 / 源码预览”
- HTML 支持“页面预览 / 源码预览”
- HTML 页面预览优先服务静态内容，不承诺脚本驱动页面完全可交互
- 相对资源尽量按工作空间路径解析
- 页面预览失败时可以明确回退到源码预览
- 右侧面板改为可拖拽调宽抽屉
- 每次弹出抽屉都回到统一默认宽度，不记忆上次用户拖拽结果
- 文件名显示时保留后缀，悬停可看到完整文件名和相对路径

## 非目标

本次不做：

- 新建独立的本地 HTTP 预览服务
- 完整还原所有 JS 驱动页面行为
- 富文档渲染器（如 `.docx` 富文本预览）
- 新的文件管理信息架构
- 会话外文件历史能力
- 关闭后记忆抽屉宽度

## 现状分析

### 已有能力

前端已有：

- 文件面板和文件树
- 文件搜索
- 文件选择后的预览请求
- Markdown 渲染预览
- HTML iframe 页面预览
- HTML / Markdown 的源码模式切换

后端已有：

- 工作空间递归枚举
- 按扩展名识别 `markdown`、`html`、`text`、`binary`、`docx`
- 文件内容读取
- 最大预览大小限制 `256 KB`

测试已有：

- 前端文件面板预览测试
- 聊天侧边栏集成测试
- Rust 侧预览命令测试

### 当前主要缺口

#### 1. HTML 相对资源解析不稳

当前 HTML 预览通过 `srcDoc` 注入 `base href`，但 `base href` 的拼接逻辑偏脆弱，容易在引用相对 CSS、图片、脚本时丢失工作空间根路径或目录层级。

#### 2. HTML 预览隔离不足

当前 iframe 预览未明确加上 `sandbox` 策略。对 AI 生成的 HTML 来说，这会让页面预览的安全边界不够清晰。

#### 3. 截断和失败没有明显用户反馈

后端对大文件会截断读取，但前端没有提示“内容已截断”。HTML 如果渲染异常，也缺少“预览失败，已回退源码”的清晰反馈。

#### 4. 首版边界没有在产品层明确

如果不提前定义边界，用户可能会把 HTML 预览理解成“完整浏览器环境”，从而对脚本执行、跨资源加载和动态页面行为产生过高预期。

#### 5. 右侧面板过窄，预览区几乎不可用

当前右侧面板宽度固定且偏窄，内部又是“文件树 + 预览区”双栏布局，导致真正留给 Markdown / HTML 预览的空间不足，难以承担日常阅读需求。

#### 6. 长文件名可读性差

当前文件项直接对整个文件名做截断处理，容易把扩展名一起截掉。结果是：

- 用户无法快速判断文件类型
- 很难区分多个长文件名的相似文件
- 鼠标悬停时也看不到完整文件名和相对路径

## 方案比较

### 方案 A：在现有链路上补强预览能力 `推荐`

做法：

- 保留现有 `WorkspaceFilesPanel` 和 `FilePreviewPane`
- 保留现有 Tauri 文件读取命令
- 将固定窄侧栏升级为可调宽抽屉
- 修正 HTML 基址注入
- 为 iframe 增加安全隔离
- 改善文件列表项的信息结构与长文件名展示
- 补充预览失败、空内容、内容截断等反馈

优点：

- 改动最小
- 与现有测试和架构保持一致
- 最快形成稳定首版
- 能同时解决“预览空间不足”和“文件名不可读”两个高频问题

缺点：

- HTML 页面预览能力仍然偏“轻量”
- 对复杂资源依赖和动态页面支持有限

### 方案 B：为 HTML 建立专用本地预览服务

做法：

- 启一个仅供本地使用的静态资源服务
- iframe 指向本地服务 URL
- 将 HTML 和相对资源通过服务端路径暴露

优点：

- 相对资源解析更自然
- 更接近浏览器打开本地网页的真实效果

缺点：

- 后端复杂度明显提高
- 需要处理端口、安全、生命周期和跨平台兼容
- 对“先实现 md/html 预览”来说投入过大

### 方案 C：HTML 首版只做源码预览，页面预览跳系统浏览器

做法：

- Markdown 保持本地渲染
- HTML 在面板内只显示源码
- 页面效果交给“打开文件”或系统浏览器

优点：

- 风险最小
- 安全和还原度争议都最少

缺点：

- 用户感知较弱
- 无法满足“在面板内直接预览 HTML”的直觉需求
- 也没有解决当前抽屉空间和文件列表可读性问题

## 推荐方案

采用方案 A。

原因：

- 仓库里已经有可用雏形，不值得另起一套体系
- 需求是“先实现 md/html 预览能力”，不是“做完整网页运行容器”
- 通过补强现有链路，可以在较低风险下交付一个清晰的 v1

## 最终设计

### 1. 文件类型策略

首版仅明确支持以下预览体验：

- `.md` / `.markdown`
  - 默认打开“渲染预览”
  - 用户可切换到“源码预览”
- `.html` / `.htm`
  - 默认打开“页面预览”
  - 用户可切换到“源码预览”
- 其他文本类文件
  - 保持源码预览

### 2. 抽屉与宽度策略

右侧“面板”从固定窄侧栏升级为可调宽抽屉。

交互定义：

- 点击 `面板` 打开右侧抽屉
- 抽屉每次打开都回到统一默认宽度
- 用户可在当前打开周期内拖拽调整宽度
- 关闭后不保留本次宽度，下次重新回到默认值

宽度建议：

- 默认宽度：约 `760px`
- 最小宽度：约 `420px`
- 最大宽度：约 `1100px`

这样可以兼顾两类场景：

- 日常快速浏览文件
- 需要认真阅读 Markdown / HTML 预览时，临时拉宽获得更大空间

### 3. 文件列表与预览区布局

进入 `文件` tab 后，抽屉内部保持左右双栏：

- 左侧：文件树 / 搜索 / 文件列表
- 右侧：预览区

布局原则：

- 左侧列表区宽度保持相对稳定，优先保证浏览效率
- 右侧预览区占用剩余空间，作为真正的阅读区域
- 不额外弹出系统窗口作为主预览方式

### 4. 文件列表信息结构

文件列表项不再简单地对整个文件名做一刀切截断，而是改为更适合阅读的结构：

- 主文件名显示，但只截断主名
- 明确保留文件后缀
- 补充类型标签，如 `MD`、`HTML`
- 保留文件大小
- 保留 `本轮生成` 标记

悬停反馈：

- 鼠标悬停时显示完整文件名
- 同时展示完整相对路径

这样即使在较窄宽度下，用户也能快速知道：

- 这是不是 Markdown / HTML
- 当前看的到底是哪一个同名前缀文件
- 文件实际位于哪个目录

### 5. Markdown 预览设计

Markdown 继续使用现有 `ReactMarkdown + remark-gfm` 渲染。

首版原则：

- 不额外引入复杂插件
- 保持和聊天区 Markdown 渲染体验一致
- 重点保证标题、列表、表格、代码块可读

### 6. HTML 预览设计

HTML 继续使用 iframe 渲染，但补充以下约束：

- 页面预览使用 `srcDoc`
- 注入更稳定的 `<base href="...">`，优先按“当前 HTML 文件所在目录”解析相对资源
- iframe 添加 `sandbox`
- 默认不追求完整脚本交互

交互定义：

- 默认进入“页面预览”
- 用户可切换到“源码预览”
- 若页面预览失败或资源不可用，界面提示可退回源码查看

### 7. 安全边界

HTML 页面预览采用“可看、轻交互、弱执行”的策略。

建议首版 iframe 使用受限 `sandbox`：

- 允许展示页面内容
- 尽量不放开高风险能力
- 如果后续确实需要脚本驱动展示，再单独讨论白名单能力

首版安全原则：

- 预览优先于执行
- 内容展示优先于页面行为完整性

### 8. 预览反馈状态

文件预览区需要补齐以下状态：

- 未选择文件：`选择要查看的文件`
- 文件为空：`文件内容为空`
- 预览已截断：显示“仅展示前 N KB 内容”
- HTML 页面预览失败：显示提示并保留源码模式
- 二进制文件：明确提示“不支持内嵌预览，请使用系统默认应用打开”

预览区标题栏补充：

- 当前文件完整相对路径
- 必要时显示类型标签

### 9. 后端返回结构补强

现有 `WorkspaceFilePreview` 建议补充轻量元信息：

- `truncated: boolean`
- 可选 `preview_error: string | null`

这样前端不需要猜测当前内容是否完整，也不用把所有失败都统一当作“读取失败”。

### 10. 首版范围定义

v1 成功标准：

- 能在“面板 > 文件”中浏览工作空间文件
- 右侧抽屉每次打开都回到默认宽度，并允许临时拖拽调宽
- 选择 `.md` 后能正确显示渲染和源码
- 选择 `.html` 后能正确显示页面和源码
- 长文件名仍能清晰看出后缀和类型
- 悬停能看到完整文件名和相对路径
- 常见相对 CSS / 图片引用在静态 HTML 中尽量可用
- 遇到异常时用户能得到明确反馈，而不是空白区

## 测试策略

前端测试补强：

- 抽屉默认宽度正确
- 拖拽可在限定范围内调整抽屉宽度
- 关闭再打开后恢复默认宽度
- Markdown 默认进入渲染模式，且可切源码
- HTML 默认进入页面模式，且可切源码
- HTML 页面预览失败时仍能回到源码
- 截断提示和空内容提示正确展示
- 长文件名项仍能看到类型标签
- 文件项悬停时可获得完整文件名或路径提示

后端测试补强：

- `markdown/html` 返回的结构包含正确 kind
- 大文件预览会标记 `truncated`
- 路径越界继续被拒绝
- HTML / Markdown 读取行为与扩展名分类稳定

## 决策结论

本次采用“现有链路补强”的方式实现 md/html 首版预览。

用户确认的首版边界为：

- HTML 预览接受“静态预览优先”
- 不保证脚本驱动页面完全可交互
- 抽屉宽度不做持久化，每次弹出都回到默认值

这个边界能让我们优先交付高频、稳定、风险可控的文件预览体验。
