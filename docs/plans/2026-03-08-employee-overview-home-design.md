# 智能体员工总览页重构设计

## 1. 背景

当前“智能体员工”页面把总览、团队编排、员工详情、飞书连接、画像文件、长期记忆和全局设置混在同一长页里。用户进入页面后首先看到的是“拉群协作”大表单，而不是员工数量、团队数量和待处理问题，页面心智已经从“运营总览”滑成了“系统配置台”。

现有前端实现集中在 [apps/runtime/src/components/employees/EmployeeHubView.tsx](/E:/code/yzpd/skillhub/apps/runtime/src/components/employees/EmployeeHubView.tsx)，并且已经承担：

- 员工列表与员工详情
- 团队创建、模板复制与团队运行
- 飞书连接状态和配置
- AGENTS / SOUL / USER 文件只读预览
- 长期记忆治理
- 全局默认工作目录

这导致页面信息层级混乱、首屏认知负担过高、业务对象与操作对象不清晰。

## 2. 目标

P0 只解决一件事：把“智能体员工”重构为真正的总览页。

具体目标：

1. 默认进入 `总览` 视图，而不是直接进入配置视图。
2. 首屏明确展示员工数、团队数、可用状态和待处理事项。
3. 首页只承担“看状态 + 去动作”，不再展开复杂配置表单。
4. 通过页内二级导航把员工、团队、运行、设置分层承载，避免重新引入首页杂糅。
5. 统一首页命名，移除英文状态码和底层实现术语。

## 3. 非目标

P0 不包含以下内容：

1. 团队规则可视化流程图。
2. 首页内直接编辑飞书配置、画像文件或长期记忆。
3. 长期记忆和员工画像模块的深度重构。
4. 全新的多路由页面体系；P0 允许先采用页内 tab。
5. 全量视觉风格翻新；重点在信息架构和交互层级。

## 4. 页面定位

“智能体员工”页的产品定位调整为：`智能体员工运营总览页`。

页面只回答三个问题：

1. 我现在有多少员工和团队？
2. 哪些地方有问题需要处理？
3. 我下一步应该点哪里？

因此首页只做三类内容：

- 规模与健康度指标
- 待处理问题聚合
- 下钻入口

## 5. 信息架构

P0 采用页内二级导航，默认选中 `总览`：

- `总览`
- `员工`
- `团队`
- `运行`
- `设置`

### 5.1 总览

模块顺序固定为：

1. 页面头部
2. 核心指标
3. 待处理事项
4. 员工概览
5. 团队概览
6. 最近运行
7. 快捷入口

### 5.2 员工

承接：

- 员工列表
- 搜索/筛选
- 员工摘要
- 进入员工详情

### 5.3 团队

承接：

- 团队列表
- 团队创建/模板复制
- 团队摘要
- 进入团队详情或团队运行

### 5.4 运行

承接：

- 近期团队运行记录
- 运行状态
- 跳转到对应会话或详情

### 5.5 设置

承接：

- 全局默认工作目录
- 操作确认级别等全局配置

## 6. 首页模块定义

### 6.1 页面头部

- 标题：`智能体员工`
- 副标题：`查看员工、团队与运行状态`
- 主按钮：`新建员工`
- 次主按钮：`新建团队`
- 次级入口：`查看运行记录`

“智能体员工助手”入口保留，但降级为快捷入口，不再占用首屏 banner。

### 6.2 核心指标

展示 5 张指标卡：

1. `员工总数`
2. `团队总数`
3. `可用员工`
4. `运行中团队`
5. `待处理事项`

指标卡点击后直接切换到对应 tab，并自动附带筛选条件。

### 6.3 待处理事项

按用户可理解的问题聚合，而不是按底层字段聚合：

- 员工未完成基础配置
- 员工启用了连接能力但未完成连接配置
- 团队角色不完整
- 团队最近运行失败
- 默认入口员工缺失

每条事项必须带 `去处理` 或 `查看详情`。

### 6.4 员工概览

仅展示摘要，不展开详情。字段建议：

- 员工名称
- 角色定位
- 技能数
- 连接状态
- 最近活跃
- 当前状态

每条支持：

- `查看详情`
- `开始对话`

### 6.5 团队概览

仅展示摘要，不展开编排表单。字段建议：

- 团队名称
- 成员数
- 协作模式
- 入口员工/负责人
- 最近运行状态
- 最近运行时间

每条支持：

- `查看详情`
- `发起任务`

### 6.6 最近运行

展示最近 5-10 条员工/团队运行记录：

- 任务名
- 执行对象
- 状态
- 开始时间
- 耗时
- 入口操作

`最近运行` 需要稳定的数据源，不能只依赖当前页内临时状态。

### 6.7 快捷入口

只放低频但重要的入口：

- `打开员工助手`
- `导入团队模板`
- `员工管理`
- `团队管理`
- `系统设置`

## 7. 首页与二级页边界

以下内容必须移出首页：

- 团队编排大表单
- 员工详情正文
- 飞书连接详细输入框
- AGENTS / SOUL / USER 文件内容
- 长期记忆管理
- 全局默认工作目录表单

迁移后的承接关系：

- 员工详情、连接配置、画像、长期记忆 -> `员工`
- 团队成员、团队规则、团队启动 -> `团队`
- 近期运行列表 -> `运行`
- 全局默认工作目录等系统参数 -> `设置`

## 8. 命名策略

首页只允许出现业务对象、业务状态和业务动作，不直接暴露底层实现词。

需要统一替换：

- `创建协作群` -> `新建团队`
- `主员工` -> `默认入口员工`
- `拉群协作` -> `团队协作`
- `AGENTS / SOUL / USER` -> 仅在详情页显示为 `行为规范 / 思考方式 / 对外身份`
- `staged` -> `分阶段执行`
- `hard` -> `严格审核`
- `entry_only` -> `仅入口可见`

## 9. 技术方案

P0 优先采用最小改动路径：

1. 保留 [EmployeeHubView.tsx](/E:/code/yzpd/skillhub/apps/runtime/src/components/employees/EmployeeHubView.tsx) 作为容器。
2. 在容器内部引入页内 tab 状态：`overview | employees | teams | runs | settings`。
3. 将当前长页内容拆成多个子面板或子组件，首页只渲染总览模块。
4. 优先复用现有员工、团队和连接状态数据。
5. 为 `最近运行` 补一个明确的数据查询接口或统一前端读取口径。

## 10. 数据口径

- 员工总数：所有未删除员工。
- 团队总数：所有未删除团队。
- 可用员工：未停用且基础配置完整。
- 运行中团队：存在 `排队中 / 运行中 / 等待审核` 任务的团队。
- 待处理事项：按问题类型聚合后的事项条数，而不是员工/团队总数。

## 11. 验收标准

1. 首页首屏能同时看到标题、核心指标和待处理事项。
2. 首页不再出现团队编排大表单和员工详情正文。
3. 首页文案不再暴露英文状态码和底层实现术语。
4. 用户可以从指标卡和待处理事项直接跳到对应处理位置。
5. 页面主滚动只有一层，主内容区不再出现首页内多重滚动叠加。
6. 用户进入页面 5 秒内可以回答：
   - 有多少员工
   - 有多少团队
   - 当前有哪些待处理问题
