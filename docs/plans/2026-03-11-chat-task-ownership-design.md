# Chat Task Ownership Design

**背景**

当前聊天界面同时在两个区域表达“当前任务”：

- 主消息流顶部使用“任务进度”聚合卡
- 右侧面板第一栏使用“当前任务”与“当前步骤”卡片

这会导致两个问题：

1. 用户不知道哪一处才是当前真状态。
2. 初始空态和异常态容易出现冲突文案，例如同屏出现“处理中”和“已完成”。

该问题在“智能体员工助手”场景中尤为明显。用户尚未补充任何信息时，会话顶部已经展示执行态卡片，形成“系统已经开始做事”的误导感。

## 目标

1. 一个会话只能存在一个“当前任务”主状态源。
2. 主消息流遵守时间顺序，只展示事件，不展示全局聚合状态。
3. 空态、引导态、运行态、等待确认态、完成态必须严格区分。
4. “智能体员工助手”在用户未补充信息前展示引导态，而不是执行态。

## 非目标

- 本次不重做整个聊天 UI 风格。
- 本次不调整工具调用语义或后端会话模型。
- 本次不改变右侧“文件 / Web 搜索”页签结构。

## 核心原则

### 1. 单一职责

“当前任务”只能由右侧面板承担。

- 右侧面板负责持续态信息：
  - 当前状态
  - 当前步骤
  - 任务清单
  - 本轮产物
  - 搜索与动作摘要
- 主消息流不再承担“当前任务”职责。

### 2. 时间顺序

主消息流只展示按时间发生的关键信息，不展示跨整轮会话的悬浮聚合卡。

### 3. 无执行不显示执行态

在没有真实运行步骤、工具调用、确认动作之前，不允许出现以下词汇：

- 处理中
- 进行中
- 已完成
- 任务进度

## 方案对比

### 方案 A：彻底移除顶部任务卡

优点：

- 信息架构最干净
- 可以最快消除冲突

缺点：

- 主消息流缺少关键阶段回执

### 方案 B：将顶部任务卡改为关键事件卡

优点：

- 保留聊天式 agent 的过程感
- 不再与右侧“当前任务”冲突
- 最适合“员工创建”“配置确认”“结果回执”这类阶段性会话

缺点：

- 需要重新定义事件插入规则

### 方案 C：保留顶部卡但降级为极简状态条

优点：

- 改动较小

缺点：

- 仍会与右侧持续态形成竞争
- 后续容易再次膨胀

## 结论

采用方案 B。

最终职责划分：

- 右侧面板 = 唯一的持续态任务控制台
- 主消息流 = 关键事件流

## 最终交互设计

### 右侧面板

右侧保留当前结构，但语义重新收敛：

- 第一块：`当前状态`
- 第二块：`任务清单`
- 第三块：`本轮产物`
- 第四块：`搜索与动作`

其中：

- “当前状态”负责展示运行中、等待确认、等待补充、失败、完成等持续状态
- “任务清单”负责 TodoWrite 投影与结构化任务
- “本轮产物”负责文件和交付摘要
- “搜索与动作”负责 Web 搜索和其他动作统计

### 主消息流

移除常驻“任务进度”聚合卡，不再在消息列表顶部长期显示。

改为只在关键节点插入事件卡，事件卡必须出现在真实触发点之后，并在消息时间线上与普通消息同级。

允许的事件类型：

- `等待你补充关键信息`
- `等待你确认配置草案`
- `已生成配置草案`
- `已创建智能体员工：<名称>`
- `创建失败：<原因>`
- `已恢复继续执行`

禁止的事件类型：

- 泛化的“处理中”
- 无上下文的“已完成”
- 空态下的“任务进度”

### 智能体员工助手首屏

在用户尚未补充具体信息时，显示“引导态”而不是“执行态”：

- 主区显示一条轻提示：
  - `我会先问 1-2 个关键问题，再给出配置草案，确认后执行创建。`
- 不显示主消息流事件卡
- 右侧“当前状态”显示：
  - `等待收集需求`

### 状态文案约束

术语必须统一：

- `当前任务` 只出现在右侧面板
- 主消息流使用 `状态`、`结果`、`等待你确认`、`等待你补充`
- 不允许在同一时刻同时出现“处理中”和“已完成”

## 状态映射规则

### 空态

条件：

- 无运行中工具
- 无 TodoWrite 清单
- 无待确认动作
- 无本轮关键结果事件

结果：

- 主消息流不显示事件卡
- 右侧显示“等待收集需求”或“暂无执行中的步骤”

### 运行态

条件：

- 存在运行中步骤或工具

结果：

- 右侧显示“当前状态 = 处理中”
- 主消息流仅在进入新的关键阶段时插入事件，不显示持续 running 卡

### 等待确认态

条件：

- 出现 ask-user、风险确认、配置确认

结果：

- 右侧显示“等待你的确认”
- 主消息流插入一条确认事件卡

### 完成态

条件：

- 产生最终结果，例如员工创建成功、配置更新完成、交付文件生成完成

结果：

- 右侧显示“已完成”
- 主消息流插入结果事件卡，内容具体到结果对象

### 失败态

条件：

- 本轮终止于错误，且未恢复

结果：

- 右侧显示失败原因与可恢复动作
- 主消息流插入失败事件卡

## 验收标准

1. 用户未输入补充信息前，主消息流不显示“任务进度”卡。
2. 会话中只有右侧面板承担“当前任务”职责。
3. 主消息流的状态类内容都能对应到具体时间点与具体事件。
4. 空态不再出现“处理中 / 已完成”冲突文案。
5. “智能体员工助手”首屏体现“先提问、后草案、再确认、再执行”的心智模型。
